# Style Guide

Документ определяет единый стиль оформления всей проектной документации.

---

# Общие правила

Документация должна быть:

- последовательной;
- читаемой;
- лаконичной;
- проверяемой;
- независимой от технической реализации.

---

# Заголовки

Используются только заголовки Markdown.

Не допускается пропуск уровней.

---

# Стиль текста

Используется деловой русский язык.

Не допускаются:

- разговорные выражения;
- эмоциональные оценки;
- маркетинговые формулировки;
- неопределённые утверждения.

Предпочтение отдаётся коротким предложениям.

---

# Терминология

Используются только термины из GLOSSARY.md.

Не допускается использование разных слов для обозначения одной сущности.

---

# Именование файлов

Используется PascalCase.

Например

Professional_Journey.md

Domain_Model.md

Functional_Requirements.md

# Markdown

Используются:

- заголовки;
- маркированные списки;
- нумерованные списки;
- таблицы;
- блоки кода.

Не используются HTML-вставки без необходимости.

---

# Ссылки

Документы должны ссылаться друг на друга относительными путями.

---

# Изображения

Все диаграммы размещаются в каталоге diagrams.

---

# Принцип "один документ — одна задача"

Каждый документ отвечает только на один вопрос.

Product Book

Что строим?

Use Cases

Как используется система?

Functional Requirements

Что должна уметь система?

ADR

Почему принято архитектурное решение?

Decision Log

Какие продуктовые решения были приняты?

API

Как взаимодействуют компоненты?

---

# Изменение документации

Перед изменением структуры документации необходимо добавить соответствующее решение в Decision Log.

Документация является частью продукта и сопровождается наравне с исходным кодом.
