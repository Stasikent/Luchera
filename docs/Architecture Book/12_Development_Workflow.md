# Development Workflow

Development Workflow определяет правила организации процесса разработки платформы «Лучера».

Документ устанавливает единый подход к разработке, сопровождению, тестированию, документированию и выпуску новых версий платформы.

Все участники проекта должны соблюдать данные правила.

---

# Назначение

Development Workflow обеспечивает:

- единообразие разработки;
- высокое качество кода;
- воспроизводимость процесса;
- прозрачность изменений;
- сохранение архитектурной целостности платформы.

---

# Основные принципы

При разработке используются следующие принципы:

- Product First;
- Architecture First;
- Documentation First;
- Small Changes;
- Continuous Improvement;
- Automation First.

---

# Жизненный цикл изменения

Каждое изменение проходит следующие этапы:

```text
Идея
    │
    ▼
Decision Log (если изменяется продукт)
    │
    ▼
ADR (если изменяется архитектура)
    │
    ▼
Product / Architecture Book
    │
    ▼
Implementation
    │
    ▼
Testing
    │
    ▼
Code Review
    │
    ▼
Merge
    │
    ▼
Release
```

Изменения продукта и архитектуры должны быть документированы до начала реализации.

---

# Репозитории

Документация и исходный код развиваются совместно.

Рекомендуемая структура:

```text
docs/
    book/
    architecture/
    use_cases/
    templates/
    adr/

backend/

frontend/

infrastructure/
```

Документация должна обновляться одновременно с изменением архитектуры или функциональности.

---

# Управление версиями

Для управления исходным кодом используется Git.

Основная ветка содержит только стабильный код.

Новые возможности разрабатываются в отдельных ветках.

После завершения работы изменения проходят Code Review и объединяются с основной веткой.

---

# Code Review

Все изменения проходят обязательную проверку.

Во время проверки оцениваются:

- соответствие Product Book;
- соответствие Architecture Book;
- соблюдение архитектурных принципов;
- качество кода;
- тестируемость;
- читаемость.

---

# Документирование изменений

Изменения сопровождаются обновлением соответствующей документации.

При необходимости обновляются:

- Product Book;
- Architecture Book;
- Decision Log;
- ADR;
- Use Cases;
- Functional Requirements.

Документация считается частью реализации.

---

# Тестирование

Перед объединением изменений должны выполняться:

- unit-тесты;
- интеграционные тесты;
- статический анализ кода;
- проверка форматирования;
- проверка документации.

При необходимости выполняются ручные сценарии тестирования.

---

# Definition of Done

Изменение считается завершенным, если:

- реализована необходимая функциональность;
- код проходит автоматические проверки;
- тесты успешно выполнены;
- документация обновлена;
- Code Review завершен;
- архитектурные требования соблюдены.

---

# Выпуск новых версий

Каждая новая версия сопровождается:

- списком изменений;
- обновлением документации;
- миграциями базы данных (при необходимости);
- проверкой совместимости;
- обновлением Roadmap (при необходимости).

---

# Работа с техническим долгом

Технический долг фиксируется явно.

Приоритет устранения определяется:

- влиянием на архитектуру;
- влиянием на пользователей;
- рисками дальнейшего развития платформы.

---

# Использование AI при разработке

Инструменты искусственного интеллекта могут использоваться для:

- генерации кода;
- анализа архитектуры;
- подготовки документации;
- поиска ошибок;
- написания тестов.

Ответственность за корректность результата всегда остается за разработчиком.

---

# Связь с документацией

Development Workflow основан на:

- Product Book;
- Architecture Book;
- Decision Log;
- ADR.

Все процессы разработки должны соответствовать этим документам.
