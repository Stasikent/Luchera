# API Architecture

API Architecture определяет принципы взаимодействия компонентов платформы «Лучера».

Документ описывает организацию API, правила проектирования интерфейсов и взаимодействие между клиентскими приложениями и серверной частью.

Конкретные конечные точки (endpoints) описываются автоматически средствами OpenAPI и не являются частью настоящего документа.

---

# Назначение

API обеспечивает единый способ взаимодействия между:

- пользовательскими интерфейсами;
- серверной частью;
- интеллектуальными сервисами;
- внешними информационными системами.

---

# Основные принципы

При проектировании API соблюдаются следующие принципы:

- единообразие;
- предсказуемость;
- обратная совместимость;
- независимость модулей;
- безопасность;
- расширяемость.

---

# Архитектурная схема

```text
Frontend
      │
 REST / WebSocket
      │
      ▼
API Gateway
      │
      ▼
Application Layer
      │
 ┌────┼─────┬─────────┐
 ▼    ▼     ▼         ▼
Platform Knowledge Education Practice ...
```

Все клиентские приложения взаимодействуют только через API.

---

# Структура API

API организуется по модулям платформы.

Основные разделы:

- Platform API;
- Knowledge API;
- Education API;
- Practice API;
- Professional Development API;
- Career API.

Каждый модуль предоставляет собственный набор интерфейсов.

---

# Версионирование

Публичное API поддерживает версионирование.

Изменения, нарушающие совместимость, допускаются только в новой версии API.

---

# Формат данных

Обмен данными осуществляется в формате JSON.

Все запросы и ответы используют единый стиль именования и единые правила сериализации.

---

# Аутентификация

Доступ к защищённым методам API осуществляется после успешной аутентификации пользователя.

Авторизация определяется ролями и правами доступа.

---

# Ошибки

Все ошибки имеют единый формат.

Ответ должен содержать:

- код ошибки;
- описание;
- дополнительные сведения (при необходимости).

Внутренние ошибки системы не должны раскрываться пользователю.

---

# Идемпотентность

Операции чтения не изменяют состояние системы.

Повторное выполнение идемпотентных операций должно приводить к одинаковому результату.

---

# Пагинация

Для коллекций используется единый механизм постраничного вывода.

API должно поддерживать:

- ограничение количества элементов;
- смещение;
- сортировку;
- фильтрацию.

---

# Поиск

Поиск является частью каждого пользовательского модуля.

Все поисковые запросы используют единые принципы фильтрации и сортировки.

---

# Документирование

Публичные интерфейсы документируются автоматически средствами OpenAPI.

Документация должна быть доступна разработчикам без дополнительной подготовки.

---

# Интеграция

Внешние системы взаимодействуют только через публичное API.

Прямой доступ к базе данных платформы запрещён.

---

# Безопасность

Все обращения к API выполняются по защищённым каналам связи.

Проверка прав доступа выполняется для каждого запроса.

---

# Связь с документацией

API Architecture основана на:

- Product Book;
- Application Architecture;
- Data Model;
- Database Design.

Документ определяет принципы проектирования интерфейсов взаимодействия платформы.
