# System Architecture

System Architecture описывает общую структуру платформы «Лучера».

Документ определяет основные компоненты системы, их взаимодействие и границы ответственности.

Подробная реализация отдельных компонентов рассматривается в следующих главах Architecture Book.

---

# Архитектурный стиль

Первая версия платформы реализуется как модульный монолит.

Все пользовательские модули работают в рамках единого приложения, сохраняя логическую независимость.

Такой подход обеспечивает:

- простоту разработки;
- единое развертывание;
- низкие эксплуатационные затраты;
- возможность постепенного выделения сервисов при необходимости.

# Основные компоненты

## Frontend

Пользовательский веб-интерфейс платформы.

Отвечает за взаимодействие пользователя с системой.

Не содержит бизнес-логики.

---

## Backend API

Единая точка доступа ко всем возможностям платформы.

Обеспечивает:

- аутентификацию;
- авторизацию;
- выполнение бизнес-логики;
- работу с данными;
- взаимодействие с интеллектуальными сервисами.

---

## Platform

Общие сервисы системы.

Включают:

- пользователей;
- роли;
- организации;
- уведомления;
- поиск;
- файлы;
- аудит.

---

## Knowledge

Единая база знаний платформы.

Используется всеми пользовательскими модулями.

---

## Intelligence

Компонент интеллектуальной поддержки.

Используется для:

- рекомендаций;
- поиска;
- анализа;
- сопровождения пользователя.

Не является самостоятельным пользовательским модулем.

---

## Business Modules

Включают:

- Education;
- Practice;
- Professional Development;
- Career.

Каждый модуль использует общую доменную модель и сервисы Platform.

---

## PostgreSQL

Основное хранилище данных.

Содержит доменную модель платформы.

---

## Object Storage

Используется для хранения файлов:

- документов;
- изображений;
- презентаций;
- медицинских изображений;
- других объектов базы знаний.

---

# Взаимодействие компонентов

Frontend взаимодействует исключительно с Backend API.

Business Modules не взаимодействуют напрямую между собой.

Все общие сервисы предоставляются Platform.

Knowledge является единым источником профессиональной информации.

Intelligence использует данные Platform, Knowledge и Business Modules для формирования рекомендаций.

---

# Принципы развития

При дальнейшем развитии платформы допускается выделение отдельных компонентов в самостоятельные сервисы.

При этом:

- публичные API должны сохраняться;
- доменная модель не должна изменяться;
- пользовательские сценарии должны оставаться неизменными.

Таким образом архитектура поддерживает постепенную эволюцию от модульного монолита к распределённой системе при необходимости.
