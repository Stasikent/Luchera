# Application Architecture

Application Architecture определяет внутреннее устройство серверной части платформы «Лучера».

Документ описывает организацию исходного кода, архитектурные слои, взаимодействие компонентов и правила разработки.

Конкретная реализация отдельных модулей описывается в исходном коде проекта.

---

# Назначение

Application Architecture обеспечивает:

- единообразную структуру проекта;
- независимое развитие модулей;
- повторное использование компонентов;
- высокую тестируемость;
- простоту сопровождения.

---

# Архитектурный стиль

Платформа реализуется как модульный монолит.

Каждый пользовательский модуль является самостоятельным компонентом внутри единого приложения.

Модули взаимодействуют через публичные интерфейсы.

---

# Общая структура проекта

```text
backend/

app/

platform/

knowledge/

education/

practice/

professional_development/

career/

shared/

infrastructure/
```

Каждый модуль имеет одинаковую внутреннюю структуру.

---

# Архитектурные слои

Каждый модуль разделяется на следующие слои.

```text
API

↓

Application

↓

Domain

↓

Infrastructure
```

---

## API

Отвечает за взаимодействие с внешним миром.

Содержит:

- маршруты;
- схемы запросов;
- схемы ответов;
- преобразование DTO.

Бизнес-логика отсутствует.

---

## Application

Содержит сценарии использования системы.

Отвечает за:

- выполнение бизнес-процессов;
- координацию компонентов;
- работу с транзакциями;
- вызов доменной модели.

---

## Domain

Содержит предметную модель приложения.

Включает:

- сущности;
- объекты-значения;
- доменные сервисы;
- бизнес-правила.

Не зависит от используемых технологий.

---

## Infrastructure

Реализует технические механизмы.

Включает:

- PostgreSQL;
- файловое хранилище;
- внешние API;
- кэширование;
- очереди сообщений.

---

# Взаимодействие слоев

```text
API
 │
 ▼
Application
 │
 ▼
Domain
 │
 ▼
Infrastructure
```

Нижние слои не должны зависеть от верхних.

---

# Общие компоненты

Компоненты общего назначения располагаются в модуле Shared.

Shared содержит:

- общие типы;
- базовые интерфейсы;
- исключения;
- утилиты;
- общие сервисы.

Shared не содержит бизнес-логики конкретных модулей.

---

# Зависимости между модулями

Разрешены зависимости только через публичные интерфейсы.

Например:

```text
Education

↓

Knowledge

↓

Platform
```

Прямое обращение к внутренней реализации другого модуля запрещается.

---

# Управление зависимостями

Создание объектов выполняется посредством Dependency Injection.

Компоненты получают зависимости через интерфейсы, а не через конкретные реализации.

---

# Работа с данными

Каждый модуль работает с собственными репозиториями.

Бизнес-логика не должна выполнять SQL-запросы напрямую.

Взаимодействие с базой данных осуществляется через слой Infrastructure.

---

# Обработка ошибок

Ошибки делятся на:

- доменные;
- прикладные;
- инфраструктурные.

Каждая категория обрабатывается на соответствующем уровне архитектуры.

---

# Тестирование

Каждый модуль должен поддерживать:

- unit-тесты;
- интеграционные тесты.

Тестирование модулей должно быть максимально независимым.

---

# Расширяемость

Добавление нового пользовательского модуля должно требовать минимальных изменений существующей архитектуры.

Каждый новый модуль реализуется по тем же архитектурным правилам, что и существующие.

---

# Связь с документацией

Application Architecture основана на:

- Product Book;
- Data Model;
- Database Design;
- Architecture Principles.

Документ определяет правила реализации серверной части платформы.
