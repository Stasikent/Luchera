# Security Architecture

Security Architecture определяет принципы обеспечения безопасности платформы «Лучера».

Документ описывает архитектурные механизмы защиты данных, пользователей и компонентов системы.

Конкретные настройки безопасности определяются инфраструктурой и исходным кодом проекта.

---

# Назначение

Архитектура безопасности должна обеспечивать:

- защиту персональных данных;
- защиту профессиональной информации;
- безопасную аутентификацию пользователей;
- разграничение доступа;
- контроль действий пользователей;
- защиту интеллектуальных сервисов;
- безопасную эксплуатацию платформы.

---

# Основные принципы

При проектировании системы используются следующие принципы:

- Security by Design;
- Least Privilege;
- Defense in Depth;
- Zero Trust;
- Secure by Default;
- Auditability.

---

# Аутентификация

Платформа поддерживает аутентификацию пользователей через единый механизм.

Поддерживаются:

- локальная учетная запись;
- OAuth2;
- OpenID Connect (в дальнейшем).

Механизм аутентификации должен быть независим от бизнес-логики приложения.

---

# Авторизация

Доступ определяется ролями пользователя.

Каждая операция должна проверять права доступа независимо от пользовательского интерфейса.

Недостаточно скрыть элемент интерфейса — проверка обязательно выполняется на сервере.

---

# Управление ролями

Один пользователь может иметь несколько ролей одновременно.

Например:

- студент;
- ординатор;
- преподаватель;
- врач;
- администратор.

Набор ролей определяется Product Book.

---

# Защита персональных данных

Персональные данные обрабатываются только в объеме, необходимом для выполнения функций платформы.

Архитектура должна предусматривать:

- минимизацию собираемых данных;
- разграничение доступа;
- журналирование операций;
- возможность удаления или обезличивания данных в соответствии с требованиями законодательства.

---

# Защита медицинских данных

Медицинские изображения и документы требуют дополнительной защиты.

При использовании их в образовательных целях должны применяться механизмы обезличивания.

Интеллектуальные сервисы получают только минимально необходимый объем данных.

---

# Защита файлов

Все пользовательские файлы хранятся в объектном хранилище.

Доступ осуществляется только через Backend API.

Прямой доступ к объектному хранилищу запрещён, если это не предусмотрено архитектурой.

---

# Безопасность Intelligence

Интеллектуальные сервисы не должны:

- изменять пользовательские данные без подтверждения;
- выполнять действия от имени пользователя;
- принимать медицинские решения;
- скрывать источники информации.

Передача данных внешним моделям должна соответствовать требованиям безопасности и конфиденциальности.

---

# Управление секретами

Пароли, ключи доступа и токены не должны храниться в исходном коде.

Используются:

- переменные окружения;
- менеджеры секретов;
- инфраструктурные механизмы хранения.

---

# Аудит

Все значимые действия пользователей фиксируются.

Журналируются:

- вход в систему;
- изменение данных;
- изменение ролей;
- действия администратора;
- использование интеллектуальных сервисов (при необходимости);
- ошибки безопасности.

Журналы аудита защищаются от несанкционированного изменения.

---

# Защита API

Все обращения к API должны:

- выполняться по защищённым каналам;
- проходить аутентификацию;
- проходить авторизацию;
- проверяться на корректность входных данных.

---

# Управление сессиями

Сессии пользователей должны:

- иметь ограниченное время жизни;
- завершаться после выхода пользователя;
- быть защищены от повторного использования.

---

# Мониторинг безопасности

Архитектура должна обеспечивать:

- обнаружение подозрительной активности;
- регистрацию попыток несанкционированного доступа;
- уведомление администраторов о критических событиях.

---

# Безопасная разработка

При разработке платформы должны соблюдаться следующие требования:

- проверка зависимостей;
- регулярное обновление библиотек;
- автоматический анализ кода;
- проверка уязвимостей;
- обязательный Code Review.

---

# Связь с документацией

Security Architecture основана на:

- Product Book;
- Architecture Principles;
- Application Architecture;
- Deployment.

Требования безопасности распространяются на все компоненты платформы.
