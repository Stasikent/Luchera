# Database Design

Database Design определяет принципы проектирования базы данных платформы «Лучера».

Документ описывает правила хранения данных, организации связей, использования индексов и обеспечения целостности данных.

Конкретные SQL-схемы, миграции и определения таблиц находятся в исходном коде проекта и не являются частью настоящего документа.

---

# Назначение

Database Design используется для:

- проектирования структуры базы данных;
- разработки Backend;
- проектирования API;
- обеспечения производительности;
- поддержки масштабирования платформы.

---

# Используемая СУБД

Основной системой управления базами данных является PostgreSQL.

Причины выбора:

- зрелость;
- надежность;
- расширяемость;
- поддержка JSONB;
- полнотекстовый поиск;
- широкий набор инструментов.

---

# Основные принципы

## Domain Driven

Структура базы данных должна соответствовать Data Model.

Таблицы отражают логические сущности предметной области.

---

## Normalization

Основные сущности проектируются в нормализованном виде.

Денормализация допускается только при наличии подтвержденной необходимости.

---

## Explicit Relations

Связи между сущностями реализуются явно.

Использование неявных связей запрещается.

---

## Immutable Identifiers

Каждая запись имеет неизменяемый уникальный идентификатор.

Идентификаторы никогда не используются повторно.

---

## Auditability

История изменений значимых данных должна сохраняться.

---

## Soft Delete

Для большинства бизнес-сущностей используется логическое удаление.

Физическое удаление допускается только для служебных данных.

---

# Организация данных

Данные разделяются по логическим областям:

- Platform;
- Knowledge;
- Education;
- Practice;
- Professional Development;
- Career.

Каждая область имеет собственный набор таблиц и минимально зависит от остальных.

---

# Хранение файлов

Файлы не сохраняются непосредственно в базе данных.

База данных содержит только:

- идентификатор;
- метаданные;
- путь к объекту;
- сведения о владельце;
- контрольную сумму.

Файлы размещаются в объектном хранилище.

---

# Медицинские изображения

Медицинские изображения хранятся отдельно от основной базы данных.

В базе данных сохраняются:

- идентификатор изображения;
- метаданные;
- связи с объектами базы знаний;
- сведения об обезличивании.

---

# Индексация

Для ускорения поиска используются:

- первичные ключи;
- внешние ключи;
- составные индексы;
- полнотекстовые индексы;
- специализированные индексы PostgreSQL.

Конкретный состав индексов определяется по результатам анализа производительности.

---

# Полнотекстовый поиск

В первой версии платформы используется PostgreSQL Full Text Search.

При масштабировании допускается переход на специализированные поисковые системы без изменения бизнес-логики.

---

# Версионность

Следующие сущности должны поддерживать хранение нескольких версий:

- Knowledge Asset;
- Professional Document;
- Work Template.

Пользователь должен иметь возможность просматривать историю изменений.

---

# Миграции

Изменение структуры базы данных выполняется исключительно посредством миграций.

Каждая миграция должна быть:

- обратимой (если это возможно);
- документированной;
- проверенной автоматическими тестами.

---

# Производительность

При проектировании базы данных должны учитываться:

- объем базы знаний;
- рост количества пользователей;
- увеличение числа образовательных программ;
- расширение количества медицинских специальностей.

Архитектура базы данных не должна ограничивать дальнейшее развитие платформы.

---

# Резервное копирование

База данных должна поддерживать:

- регулярное резервное копирование;
- восстановление после сбоев;
- контроль целостности резервных копий.

---

# Связь с документацией

Database Design разрабатывается на основании:

- Product Book;
- Data Model;
- Architecture Principles.

Документ используется при реализации Backend Architecture и проектировании API.
